- [1] **Prevent build from degrading command files**
  The /mill-build full rebuild regenerates command files from skill-commands.md spec, but the build agent paraphrases instead of preserving carefully-tuned output. Observed regressions: (1) numbered sequential steps replaced with flat bullets losing ordering, (2) combined commands (do-commit, finalize-do, finalize-do-commit, finalize-do-all) lost explicit script invocations and replaced them with vague 'Run do' references that the executor cannot resolve since it only sees its own command file, (3) dropped ## Rules sections and @taskmill skill references the executor needs, (4) list.md got hardcoded example numbers instead of placeholder N, (5) descriptions shortened losing clarity. Root cause: BUILD.md says 'implement according to the behavioral spec' for scripts but lacks equivalent guidance for commands — the build agent has no rule saying 'preserve existing command output structure'. Investigate: check git log for previous rebuilds (grep for 'mill-build' or 'rebuild' in commit messages) to see if earlier builds had the same drift problem. Solution options: (a) add a 'command generation rules' section to BUILD.md specifying numbered steps, explicit script paths, skill references, and ## Rules sections, (b) store golden command files as the source of truth instead of generating from spec, (c) add a diff-check post-build step that flags regressions
  - analysis: The core tension is that command files are LLM prompt text, not compiled code — phrasing, structure, and specificity affect execution quality. There is no mechanical transformation from spec to output; each rebuild is a creative act by a different LLM context. Skills are already copied verbatim (source = output). Scripts benefit from generation (Python code is mechanically verifiable). Commands sit in between — they accumulate implementation-specific detail (explicit script paths, numbered step ordering, Rules sections, skill references) that the behavioral spec in skill-commands.md does not capture. Three options: (A) Make commands source files in doc/, copied verbatim like skills — skill-commands.md becomes summary/index only. (B) Make the spec so detailed it includes exact structure — but then spec = output, defeating the purpose. (C) Hybrid: generate scripts, copy commands and skills verbatim. Option A/C maps to a real distinction: scripts have verifiable correctness (imports, exit codes), commands have subjective quality (phrasing, completeness). Different things need different build processes.
  - migration: Commands are deprecated in Claude Code — skills are the replacement. Skills support model frontmatter, auto-triggering, allowed-tools, context:fork, and progressive disclosure. Converting commands to skills solves the degradation problem entirely: (1) Create doc/taskflow/commands/skill-<name>.md for each command as a proper skill source file with full YAML frontmatter. (2) Build copies them verbatim to build/taskmill/skills/<name>/SKILL.md — no LLM generation step, no paraphrasing, no drift. (3) skill-commands.md becomes documentation/index only, or gets deleted. (4) This aligns commands with the existing skill pipeline (already verbatim-copied). (5) Gains skill features: disable-model-invocation for user-only commands, allowed-tools restrictions, context:fork for isolation. The build output changes from commands/ to skills/ for these files. Check if existing skill names conflict with command names (e.g. both 'workflow' skill and 'do' command become skills).
